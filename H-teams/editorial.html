Teams

Teams is solvable in O(n2Ka)O(n^2K_a)O(n2Ka​).

This is about n2Ka=107n^2K_a = 10^7n2Ka​=107 operations.

First, we always want to simplify the problem as much as possible. What can we remove without losing information?

Well, we can merge friends into one, as if they were one person. Friends are guaranteed to be in the same final team, so we are really adding these “friend groups” to teams.

If we imagine the players as nodes and friendships as edges (which should be a reasonable starting point), a friend group represents a connected component of the undirected graph. These connected components can be found in the standard manner: iterating over each node and starting a DFS at each unvisited node.

How do we maintain skill data in friend groups?

We can let the “skill” of a friend group be equal to the sum of the skills of the players in that group.

Furthermore, since the problem requires the groups have equal player counts and requests information about the players, we must also give each friend group a “player count” property.

This is a definition for a friend group, in C++.

struct friend_group {
        int skill;
        vector<int> players;
};

Let nfn_fnf​ be the number of friend groups.

So, we have nfn_fnf​ friend groups, and we need to partition these friend groups into two teams, each with n2\frac{n}{2}2n​ players.

Focusing on Team A, we need to choose friend groups that will not have a total skill over KaK_aKa​, but have a total player count of n2\frac{n}{2}2n​.

Do you see the next step?

Yes, we now use Knapsack DP. The team is our “knapsack”, the friend groups are our “stones”, the skills are our “profits”, the player counts are our “weights”, and KaK_aKa​ is our maximum weight!

YAY!

The transitions are somewhat-standard for Knapsack DP. First, we solve a slightly simpler problem:

Consider the choices we have if we are finding dp[i][v][w]dp[i][v][w]dp[i][v][w]. In the general case, we can:

  • Add the iii-th friend group. Define the skill of the iii-th friend group as viv_ivi​; the weight, wiw_iwi​. Define v′=v−viv' = v - v_iv′=v−vi​ and w′=w−wiw' = w - w_iw′=w−wi​. If and only if either option is available, then dp[i][v][w]dp[i][v][w]dp[i][v][w] is reachable therefore true.

We are so close to solving the problem! We only need to find a combination of assignments that lead to a correct answer. We need to maintain the last added friend group in our DP states. To do this:

  • For each time our DP chooses to add the iii-th friend group, we set dp[i][v][w]dp[i][v][w]dp[i][v][w] to point to dp[i−1][v′][w′]dp[i-1][v'][w']dp[i−1][v′][w′].
  • For each time our DP chooses to not add the iii-th friend group, we set dp[i][v][w]dp[i][v][w]dp[i][v][w] to point to what dp[i−1][v][w]dp[i-1][v][w]dp[i−1][v][w] points to.

As a result, dp[i][v][w]dp[i][v][w]dp[i][v][w] points to the last state where a friend group was added in a combination to achieve exactly vvv profit and www weight using only the first iii friend groups (in any consistent order).

Our base case is that dp[0][0][0]dp[0][0][0]dp[0][0][0] exists and equals some “Origin” value. All other dp[i][v][w]dp[i][v][w]dp[i][v][w] are initialized to some “Nonexistent”/“DNE” value (you have to determine ways to represent Origin and DNE).

Now, once we iterate and find a working dp[i][n2][w]dp[i][\frac{n}{2}][w]dp[i][2n​][w] state, we only need to iterate “backwards” by travelling from each state to the pointed state, until we inevitably reach the origin, and save all the players of the friend groups of the visited states to Team A.
